# 功能需求 N7：Chat 页面消息模型与展示方案

## 背景与问题
当前 chat 页面已经完成第一版结构化消息改造，用户、助手、系统和错误消息已不再以单纯文本行直出，而是以消息块形式进入时间线；历史 session 恢复也已切换到同一套消息视图模型。

但从完整需求角度看，这一版仍是基础落地，距离最终目标还有明显缺口：
- 当前 UI 已从 `LineItem[]` 平铺文本迁移到结构化消息模型，但消息角色仍较少，主要覆盖 `user`、`assistant`、`system`、`error`。
- `thinking` 已经从正文中剥离，并以默认折叠的块显示，但仍缺少更明确的展开交互和精细控制。
- 工具过程的数据结构已预留，但尚未真正接入摘要展示，暂时没有完整的 `toolSummary` 内容来源。
- 会话顶部信息和时间线信息已经分层，但消息级元信息与轮次摘要仍然偏轻量，后续仍可增强。

因此，N7 既记录本轮已实现的展示方向，也定义 chat 页面下一阶段的完善边界，避免再次退回到纯日志视图。

## 目标
- 将 chat 页面从平铺日志流升级为结构化消息流。
- 明确用户消息、助手消息、系统消息、错误、`thinking` 和工具过程的展示边界。
- 让最终回答成为主阅读对象，同时保留 `thinking` 与工具过程的可见性但降低打扰。
- 保证历史 session 恢复后，chat 展示模型与实时对话展示模型保持一致。

## 非目标
- 本阶段不修改 session 存储 schema，持久化仍以原始 `AgentMessage[]` 为主。
- 本阶段不实现完整的工具事件树或逐条工具日志时间线。
- 本阶段不引入独立工具面板或二级详情页。
- 本阶段不要求完整复刻 ChatGPT 式交互或 Web 聊天产品的全部视觉细节。

## 设计要点
### 结构化消息模型
chat 页面应使用独立的 UI 消息模型，而不是退回到平铺文本流。最小角色集合应包括：
- `user`
- `assistant`
- `system`
- `thinking`
- `toolSummary`
- `error`

当前已实现的 UI 模型用于承载消息级展示语义，不要求与 session 持久化结构完全一致。UI 层继续从原始 `AgentMessage[]` 与运行时事件中派生出适合渲染的消息结构。

### 用户消息展示
- 用户消息显示“正文 + 简头”。
- 简头只包含轻量信息，例如角色标识、发送时间、可选状态。
- 用户消息不显示模型、token 或其他技术性元信息。

### 助手消息展示
- 助手消息以最终回答正文为主。
- 助手正文应作为单独的主块展示，不与 `thinking`、错误、系统提示混排。
- 流式返回时，文本增量应持续归并到同一条助手消息块，而不是拆成多条日志。

### Thinking 展示
- `thinking` 默认折叠，不直接混入正文。
- 默认状态下仅展示一个弱化的折叠入口，例如“思考过程”或“Reasoning”。
- 展开后再查看详细 `thinking` 内容，确保主阅读路径优先看到最终回答。
- 当前第一版已完成“从正文中剥离并默认折叠”的能力，后续重点是补交互和信息密度控制。

### 工具过程展示
- 工具过程以折叠摘要展示，而不是全量日志刷屏。
- 第一阶段只要求摘要级表达，例如“读取 3 个文件，执行 1 个命令”。
- 后续如果需要更细粒度追踪，可在此基础上扩展，但本需求不要求先做完整工具事件树。
- 当前仅完成数据结构预留，尚未接入真实工具摘要块，这仍属于后续实现项。

### 系统与错误展示
- 系统提示和错误以独立系统块进入时间线，不并入助手正文。
- 系统块应与用户/助手消息在视觉上区分，但仍保留在上下文中可见。
- 错误块需要比普通系统块更强的警示性，以便用户快速识别失败原因。

### 会话级与消息级信息分层
- session header 只保留会话级信息，例如标题、session id、状态、更新时间、消息数。
- 时间线承担消息级展示，不再依赖顶部区域承担消息细节。
- 历史恢复后，至少保证用户消息和助手消息正文能够正确还原到相同展示模型中。

## 用户可见行为
- 用户发送消息后，chat 中出现一个清晰的用户消息块，包含正文和轻量头部。
- 助手回复时，正文在同一个助手消息块中流式增长，而不是不断追加碎片行。
- 若模型产生 `thinking` 内容，用户默认看到的是折叠入口，而不是与正文混杂的长文本。
- 若运行中存在工具过程，用户默认看到的是简短折叠摘要，而不是整个事件流滚屏。
- slash 命令反馈、恢复失败提示、未知命令等内容以独立系统块出现在时间线上。
- 错误消息以独立错误块展示，和普通系统提示有所区别。
- 恢复历史 session 时，chat 页面以同样的结构化消息方式展示正文内容，而不是退化成纯文本流水。

## 验收标准
- Given 用户发送一条普通消息，When 消息进入 chat 时间线，Then 页面应显示一个结构化用户消息块，包含正文和轻量头部。
- Given 助手正在流式回复，When 持续收到文本增量，Then 增量应归并到同一条助手消息块，而不是生成多条碎片日志。
- Given 模型返回 `thinking` 内容，When 页面渲染该轮消息，Then `thinking` 默认应以折叠块呈现，而不是与正文混排。
- Given 一轮消息中存在工具调用过程，When 页面展示该轮内容，Then 工具过程应显示为折叠摘要，而不是全量日志滚动。
- Given 发生系统提示或错误，When 该信息进入 chat，Then 应显示为独立系统块或错误块，而不是并入助手正文。
- Given 恢复一个已有 session，When chat 页面展示历史消息，Then 用户与助手消息的顺序和正文显示应正确。
- Given 当前没有任何消息，When 进入 chat 页面，Then 页面应保留空态提示。
- Given 存在长回复或多轮会话，When 用户浏览时间线，Then 页面可读性应保持稳定，不退化为难以阅读的纯日志堆叠。

## 风险与回滚
- 风险：如果 UI 继续沿用 `LineItem[]` 追加式文本流，后续接入 `thinking`、工具摘要和系统块时会不断堆积条件分支，难以维护。
- 风险：如果直接把持久化消息结构当成 UI 结构使用，容易把运行时语义、存储语义和展示语义耦在一起。
- 缓解：在 UI 层引入独立消息模型，由运行时事件和 session 数据派生出稳定的展示结构。
- 回滚：若结构化消息模型实现过程引入问题，可暂时退回到正文优先显示，但仍应保留本需求文档定义的展示边界与验收标准，避免再次滑回纯日志方案。

## 里程碑
- M1：定义 chat 页结构化消息模型与角色集合。
- M2：完成用户、助手、系统、错误的基础消息块展示。
- M3：将 `thinking` 与工具过程从正文中剥离，改为折叠块/折叠摘要。
- M4：完成历史 session 恢复到结构化消息视图的适配。
