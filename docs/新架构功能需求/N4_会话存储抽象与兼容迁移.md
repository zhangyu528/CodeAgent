# 功能需求 N4：会话存储抽象与兼容迁移

## 背景与问题
N3 解决了生命周期与持久化基线，但当前实现仍耦合在具体 JSON 文件结构，存在以下扩展瓶颈：
- 存储实现与业务语义耦合，难以替换后端（如 SQLite）。
- 缺少 schema 版本与迁移机制，长期演进风险高。
- 多进程/多实例并发写入冲突没有标准处理策略。

## 目标
- 定义稳定的 `SessionRepository` 抽象接口。
- 保持 JSON 实现为默认路径，同时预留 SQLite 实现接入能力。
- 引入 schema version 与迁移机制，保证向后兼容读取。
- 明确并发冲突处理策略，避免会话覆盖和数据错乱。

## 非目标
- 本阶段不强制切换默认存储后端。
- 本阶段不实现全文检索、语义检索能力。
- 本阶段不引入跨机器分布式存储。

## 设计要点
### Repository 抽象
最小接口：
- `create(initialMeta?)`
- `appendMessages(sessionId, messages)`
- `get(sessionId)`
- `listRecent(limit, cursor?)`
- `updateMeta(sessionId, patch)`
- `markStatus(sessionId, status)`

### 双实现路径
- `JsonSessionRepository`：默认实现，兼容现有会话文件。
- `SqliteSessionRepository`：可选实现，提供更强查询能力。
- 通过配置开关选择后端，UI 层不直接依赖具体存储。

### 版本与迁移
- 会话文档增加 `version`。
- 读取时执行迁移链：`v1 -> v2 -> ... -> latest`。
- 迁移失败时降级为“只读恢复”，并提示用户执行修复。

### 并发写保护
- 乐观并发：元数据包含 `revision`（或 `updatedAt`）进行 compare-and-set。
- 冲突时返回明确错误并触发上层重试策略。
- 对 JSON 文件路径增加文件级锁或单进程写队列策略。

## 用户可见行为
- 用户不感知存储后端变化。
- 历史与恢复行为保持一致；老会话可正常读取。
- 冲突场景下给出可理解提示，不 silent fail。

## 验收标准
- Given 历史 v1 会话文件，When 升级后读取，Then 可自动迁移并正常恢复。
- Given 同一会话并发写入，When 发生冲突，Then 不会静默覆盖且有可观测错误。
- Given 切换 JSON/SQLite 后端，When 执行 `/history` 与 `/resume`，Then 行为一致。

## 风险与回滚
- 风险：迁移链 bug 造成会话不可读。
- 缓解：迁移前保留快照；迁移失败回退到旧版本读取路径。
- 回滚：支持禁用迁移与后端切换，回到 JSON 默认实现。

## 里程碑
- M1：Repository 接口与 JSON 实现重构。
- M2：schema version + migration 框架上线。
- M3：SQLite 实现接入并通过一致性对比测试。
