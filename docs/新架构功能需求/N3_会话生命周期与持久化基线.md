# 功能需求 N3：会话生命周期与持久化基线

## 背景与问题
当前会话能力已具备基础可用性（`/history`、`/resume`、会话落盘），但仍存在以下结构性问题：
- 会话生命周期语义依赖 UI 侧流程，缺少统一状态定义。
- 已改为异步 JSON 写入，但会话创建时机、标题生成和 UI 展示语义需要进一步固定。
- 原子写入链路已落地，仍需通过需求文档明确 `.tmp -> rename` 与异常清理策略。
- 当前会话切换命令语义已收敛到 `/new`、`/history`、`/resume`，需要明确 `/clear` 仅保留清屏语义。
- history 已具备标题、更新时间、状态等元数据，但后续治理能力（归档、筛选、统计）仍缺完整边界定义。

## 目标
- 建立统一且可测试的会话生命周期状态模型。
- 明确 `/new`、`/resume`、`/history` 的会话语义与行为边界。
- 固定“首条普通消息创建 session”的触发时机与标题规则。
- 将会话存储升级为异步且具备原子写入保障。
- 定义最小可用会话元数据集合，支撑后续扩展。

## 非目标
- 本阶段不切换到 SQLite。
- 本阶段不实现会话删除/归档/导出导入。
- 本阶段不引入复杂检索与摘要能力。

## 设计要点
### 生命周期状态
- `active`：当前正在接收新消息。
- `completed`：一次任务链路完成，允许后续继续追加。
- `interrupted`：被用户中断（如 Ctrl+C）或运行中断。
- `error`：当前轮次因异常失败，保留可恢复上下文。

### 最小元数据模型
`SessionMeta` 必须至少包含：
- `id`
- `title`
- `updatedAt`
- `messageCount`
- `model`
- `provider`
- `status`
- `version`

### 存储基线
- 写入策略：`tmp file -> fsync -> rename`（原子替换）。
- IO 策略：全部改为异步路径，避免主交互线程阻塞。
- 容错策略：读取损坏文件时跳过并记录错误，不影响可用会话列表。
- 临时文件策略：启动时清理历史 `.tmp` 残留，写入失败时确保临时文件被删除。

## 用户可见行为
- `/new`：仅结束当前上下文并回到 welcome；不预创建 `sessionId`。
- 首条普通消息：创建 `sessionId`、进入 chat、以首条用户消息生成 title。
- chat 顶部：展示当前 session 的 `title / 短 id / status / messageCount / updatedAt`。
- `/history`：以弹窗列表方式展示最近会话元数据（至少标题、更新时间、状态、消息数）。
- `/resume`：恢复最近 session；若不可恢复，给出明确提示并保持当前输入上下文。
- `/clear`：仅清空当前 chat 视图，不切换 session、不创建新 session。

## 验收标准
- Given welcome 页面且当前无 `sessionId`，When 用户发送第一条普通消息，Then 才创建 session 并进入 chat。
- Given 用户连续多轮对话，When 每轮结束自动保存，Then 同一对话只更新同一个 session 文件，重启后可完整恢复。
- Given 用户执行 `/new`，When 未发送新消息前查看会话目录，Then 不应产生新的空白 session 文件。
- Given 用户执行 `/new`，When 再执行 `/resume`，Then 能恢复到切换前旧会话而非空白会话。
- Given 写入中断（模拟进程 kill），When 下次读取，Then 不应出现半写 JSON 导致整体历史不可用。
- Given 用户执行 `/history`，When 弹出会话列表并选择某一项，Then 应恢复对应 session，且 chat 顶部信息与 history 元数据一致。
- Given 大会话（消息数高）场景，When 保存与读取，Then 交互不卡顿且命令响应在可接受范围。

## 风险与回滚
- 风险：异步写入时序导致 UI 显示与持久化短暂不一致。
- 缓解：将“待持久化”与“已持久化”状态分离，失败时可重试。
- 回滚：保留当前 JSON 格式兼容读取，必要时退回同步写入开关。

## 里程碑
- M1：生命周期状态与元数据字段定义落地。
- M2：异步+原子写入链路落地并通过异常中断测试。
- M3：CLI 命令语义与 UI 展示对齐（`/new`、`/history`、`/resume`、chat header）。
