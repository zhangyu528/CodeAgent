## 第一阶段：Agent Core (MVP 基础版)

本阶段的目标是建立 Coding Agent 的“骨架”，即能跑通大模型调用并执行本地工具的具体回路。整个第一阶段完全基于 **Node.js + TypeScript** 进行构建。

### 包含的开发模块 (对应 P0 开发优先级)

第一阶段的核心开发完全对应于 **P0 核心基础模块** 中规划的内容，具体包含以下三大核心模块：

- **LLM Engine (大模型引擎)**
  - **(已实现)** 采用 **Provider 模式** 架构，建立统一的 `LLMProvider` 和 `LLMEngine` 接口。
  - **(已实现)** 完成至少一个 LLM Provider 的接入 (目前已实现 **Zhipu AI / GLM Provider** 直连 API)。
  - **(已实现)** 支持普通消息对话、传递 JSON Schema 工具定义并解析工具调用请求。

- **Tool System (工具系统)**
  - **(已实现)** 基于 `zod` 库建立标准化的 TypeScript 强类型工具接口。
  - **(已实现)** 完成核心架构，能将 Zod 的参数定义自动转为 LLM 需要的 Function Calling JSON Schema。
  - **(已实现)** 实现少数基本工具 (目前已实现读取文件 `ReadFileTool` 和回显测试 `EchoTool`) 作为 MVP 验证。

- **Agent Controller (流程控制器)**
  - **(已实现)** 实现核心的 Agent 执行循环 (Agent Loop)。
  - **(已实现)** 跑通了 `LLM -> 发起工具调用请求 -> 解析并执行 TypeScript Tool -> 拿到结果回传给 LLM -> LLM 输出终态答案` 闭环。
  - **(已实现)** 建立独立的系统提示词引擎 (System Prompt)，将运行时环境信息 (OS/Path)、核心规则规范和 ReAct 工作流约束等动态组装并注入到大模型上下文中。
  - **(已实现)** 加入 `Max Steps Limit` 硬性拦截防死循环。
  - **(已实现)** 基于 Node.js `EventEmitter` 构建了完整的状态观测总线 (抛出 thought, action, observation 节点事件)。

- **Prompt Engine (提示词引擎)**
  - **(已实现)** 建立独立的系统提示词管理模块 (`src/prompts/system_prompt.ts`)。
  - **(已实现)** 实现动态环境上下文注入（OS、架构、当前工作目录等）。
  - **(已实现)** 定义了 ReAct 工作流规范、行为准则及死循环求援机制。

- **Engineering Infrastructure (工程基础设施)**
  - **(已实现)** 初始化 Node.js + TypeScript 开发环境及代码规范配置。
  - **(已实现)** 建立 `feat/p0-core-mvp` 开发分支及核心目录结构 (`src/llm`, `src/tools`, `src/controller`, `src/prompts`)。
  - **(已实现)** 实现基于 `.env` 的多环境变量管理与运行时动态校验。

- **System Verification (系统验证)**
  - **(已实现)** 编写 `src/test-runner.ts` 自动化测试脚本，实现对 Provider、Tool 和 Controller 的端到端整合测试。
  - **(已实现)** 完成首个核心路径验证：通过 GLM 模型自主调用 `ReadFileTool` 和 `EchoTool` 完成复杂任务并输出总结。

### 系统能力与验收标准

系统在此阶段应具备以下最小能力：

**LLM → Tool → Result**

可以通过编写一个简单的测试脚本或入口文件，向 Agent 输入自然语言指令（例如：“读取 index.ts 的前 10 行并总结”），系统能够：
1. 正确调用大模型并将其意图转化为读文件工具的调用。
2. 在本地正确读取该文件。
3. 将读取结果交回大模型，并由大模型最终输出总结文本。
