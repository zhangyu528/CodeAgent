# P1：执行能力模块详细设计 (Node.js + TypeScript)

在凭借 P0 核心模块跑通基本的 LLM 调用与工具执行回路后，P1 阶段的核心目标是赋予 Agent 真实影响项目环境的“手脚”以及初步的“大脑”。这主要由 Execution Environment 和 Planner 两个模块构成。

## 1. Execution Environment (执行环境)

使 Agent 能够在真实的操作系统环境中安全、稳定地运行 Shell 命令、测试脚本或甚至动态编译代码。

- **核心职责**：
  - 封装底层的操作系统调用接口。
  - 管理子进程的生命周期 (启动、挂起、终止)。
  - 捕获子进程的 `stdout` 和 `stderr`，并在超时或输出过长时进行截断或报警。
  - 为未来的 Security Layer (P2) 预留路径检查和命令拦截拦截点。

- **技术选型**：
  - **Node.js `child_process` 模块**：利用 `spawn` 和 `exec` 方法执行外部进程。`spawn` 适合处理长时间运行且有大量流式输出的进程，而 `exec` 适合短平快的简单命令。
  - **Google `zx` 库 (可选)**：如果需要编写更复杂的跨平台执行逻辑，`zx` 可以极大地简化由于不同系统 (Windows/Linux) 命令差异带来的痛苦，它允许直接使用方便的 Markdown 式模板字符串来执行脚本。
  - **接口定义 (TypeScript)**：
    ```typescript
    export interface ExecutionResult {
      exitCode: number;
      stdout: string;
      stderr: string;
      timeout: boolean;
    }

    export interface ExecutionContext {
      cwd: string;
      env: Record<string, string>;
      timeoutMs?: number;
    }

    export class ExecutionEnvironment {
      // 执行普通的 Shell 命令
      async executeCommand(command: string, context?: ExecutionContext): Promise<ExecutionResult> { ... }
      
      // 执行特定的源文件 (例如执行一个新生成的 .ts 或 .py 脚本)
      async executeScript(filePath: string, context?: ExecutionContext): Promise<ExecutionResult> { ... }
    }
    ```

## 2. Planner (任务规划器)

在 Agent 需要处理复杂任务（如：“帮我把项目里遗留的 .js 文件全部重构为 .ts 并解决报错”）时，单次模型调用或纯粹的 ReAct 循环往往容易陷入死胡同或上下文爆炸。Planner 的引入可以进行降维打击，将大目标拆解为可管理的步骤。

- **核心职责**：
  - 分析用户的复杂高优指令。
  - 将大目标 (Objective) 拆解为顺序或并行的任务子集 (Task List)。
  - 将每个任务子集转化为针对 P0 控制器的一次或多次微小调用。
  - 跟踪任务步骤的执行状态（Pending, InProgress, Completed, Failed）。

- **技术选型**：
  - **基于 LLM 的启发式生成**：让模型首先生成一份 JSON 格式或特定 Markdown 格式的 Task List，然后我们在 Node.js 侧解析它。
  - **DAG (有向无环图) 依赖编排 (可选进阶)**：对于复杂的并行任务，可以引入轻量级的 DAG 库或自行实现基于 Promise 的并发控制器，从而并行执行彼此独立的任务。
  - **接口定义 (TypeScript)**：
    ```typescript
    export interface PlanStep {
      id: string;
      description: string;
      dependsOn?: string[];
      status: 'pending' | 'running' | 'completed' | 'failed';
    }

    export interface Plan {
      objective: string;
      steps: PlanStep[];
    }

    export class Planner {
      private engine: LLMEngine; // 复用 P0 的 Engine 单独用于规划推理
      
      // 让 LLM 生成执行计划
      async createPlan(objective: string): Promise<Plan> { ... }
      
      // 按顺序/DAG 控制执行计划，底层调用 Agent Controller 完成单个 Step
      async executePlan(plan: Plan, controller: AgentController): Promise<void> { ... }
    }
    ```

## 总结

P1 阶段引入了 Execution Environment 作为与外部真实世界互动的桥梁，使得 Agent “能做事” (如运行测试，执行编译)；同时引入了 Planner 抽象剥离任务拆解职责，使得 Agent “能做复杂的事”。当这两个模块就绪，Coding Agent 的基本生产力价值便得以初步展现。
