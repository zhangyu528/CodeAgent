# P0：核心基础模块详细设计 (Node.js + TypeScript)

在确立了 Node.js 和 TypeScript 的技术栈方向后，P0 阶段的核心目标是跑通最小的 Agent 循环：**LLM → Tool → Result**。以下是针对此目标的详细模块设计。

## 1. LLM Engine (大模型引擎)

负责与底层的大语言模型 (如 GPT-4, Claude 3, DeepSeek 等) 进行通信和抽象。由于 Node.js 生态丰富，我们可以采取轻量级的封装架构。

- **核心职责**：
  - 维护对不同模型提供商 API 的调用接口。
  - 统一处理请求参数（Prompt, Temperature, Max Tokens 等）。
  - 处理流式输出 (Streaming) 和普通响应。
  - 处理上下文及 Token 计算限制。

- **技术选型**：
  - 采用 **Provider 模式** 进行架构设计。不直接耦合具体的官方 SDK（如 `openai` SDK），而是抽象出一套统一的 Provider 接口。这样可以非常灵活地接入不同的模型提供商（如 OpenAI, Anthropic, 本地 Ollama, 第三方中转 API 等）而无需大规模重构核心逻辑。
  - **接口定义 (TypeScript)**：
    ```typescript
    // Provider 基础抽象
    export interface LLMProvider {
      name: string;
      generate(messages: Message[], options?: GenerateOptions): Promise<LLMResponse>;
      generateStream(messages: Message[], options?: GenerateOptions): AsyncIterable<string>;
    }

    // Engine 负责管理和路由 Provider
    export class LLMEngine {
      private providers: Map<string, LLMProvider>;
      
      registerProvider(provider: LLMProvider) { ... }
      getProvider(name: string): LLMProvider { ... }
      
      // 使用特定 Provider 执行生成
      async generate(providerName: string, messages: Message[], options?: GenerateOptions) { ... }
    }
    ```

## 2. Tool System (工具系统)

基于 Node.js，工具系统可以将各种本地能力（读写文件、执行脚本）抽象给 LLM。利用 TypeScript 的强类型和 Zod 等 Schema 库，可以非常安全地定义和校验工具。

- **核心职责**：
  - 注册和管理所有的可用工具 (Tools)。
  - 在请求 LLM 时，将工具的描述 (Name, Description, Parameters Schema) 转换为大模型支持的 Function Calling 格式。
  - 接收模型返回的工具调用指令 (Tool Call)，并路由执行具体的 TypeScript/JavaScript 方法。
  - 将执行结果 (Tool Result) 返回给 LLM。

- **技术选型**：
  - **参数校验**：使用 `zod` 库来定义工具参数，Zod 可以轻松地转换为 LLM 需要的 JSON Schema 格式。
  - **接口定义 (TypeScript)**：
    ```typescript
    import { z } from 'zod';

    export interface Tool<T extends z.ZodTypeAny = any> {
      name: string;
      description: string;
      parameters: T;
      execute(args: z.infer<T>): Promise<string | any>;
    }
    ```
  - **示例工具 (ReadFileTool)**：通过 Node.js 原生 `fs/promises` 模块实现读取本地文件内容。

## 3. Agent Controller (Agent 流程控制)

Controller 负责将 LLM Engine 和 Tool System 组合起来，形成一个完整的执行循环。

- **核心职责**：
  - 管理初始 System Prompt 和用户交互历史。
  - 实现 ReAct (Reason + Act) 或直接的 Function Calling 循环逻辑路线：
    1. 发起请求：向 LLMEngine 发送用户指令和所有的可用 Tool Schema。
    2. 判断意图：解析 LLM 响应，如果需要调用工具，则挂起流程。
    3. 工具调度：将解析出的参数交由 Tool System 校验并执行对应工具。
    4. 反馈闭环：将工具的执行结果追加到消息列表中，再次请求 LLMEngine。
    5. 终止条件：当 LLM 直接返回最终文本答案，且没有更多的工具调用需求时，结束循环。
  
- **技术选型**：
  - Node.js 的异步机制 (Async/Await) 非常适合编写这种等待模型响应与本地 IO 并行的编排逻辑。
  - **接口定义 (TypeScript)**：

## 总结

P0 阶段不需要过度设计复杂的 Planner 或并发模型，其核心在于：依靠 TypeScript 建立坚固的类型和契约，通过 Node.js 高效的异步能力跑通单线程的、一次性的任务流（LLM -> Tool -> Result）。一旦这三个基础模块完成，后续的扩展可以直接在这个循环骨架上添加逻辑。
